# Kimodo/BONES-SEED Motion Primitive Integration

## Objective

Replace (and expand beyond) the current CMU-mocap primitive library with production-quality, text-prompted combat motions generated by NVIDIA Kimodo on the Unitree G1 skeleton. Every new primitive must feed the existing MotionBricks VQVAE pipeline without procedural fallbacks, and must pass the same visual QA gates as the current CMU Strike/Idle primitives.

## Why Kimodo / BONES-SEED

- **Direct G1 output**: Kimodo-G1-SEED-v1 generates motion natively on the Unitree G1 skeleton, the same skeleton our MotionBricks feature pipeline already consumes after CMU retargeting.
- **Text conditioning**: We can generate distinct actions ("overhead longsword strike", "high block", "sword thrust", "evasive dodge") from natural-language prompts instead of hunting for specific mocap clips.
- **Deterministic**: Kimodo accepts a `--seed`, so generated clips are reproducible and fit the combat-truth determinism invariant.
- **License**: Kimodo code is Apache-2.0; Kimodo-G1-SEED-v1 checkpoint is NVIDIA Open Model; BONES-SEED annotations are CC BY 4.0. All are compatible with the project's redistributable-data requirement.
- **Quality**: Trained on 288 hours of publicly available mocap (BONES-SEED) / 700 hours total (Rigplay), producing higher-fidelity swordplay than the limited CMU swordplay subject alone.

## Scope

This milestone produces a new primitive library with at least six distinct actions for the Longsword / Top stance:

1. `Strike` — overhead / downward longsword strike
2. `Block` — high guard / parry
3. `Thrust` — forward sword thrust
4. `Grab` — unarmed clinch / tackle entry (prompted as close-quarters motion)
5. `Dodge` — side-step / evasive body motion
6. `Idle` — calm standing guard (replaces CMU idle or kept as variant)

Out of scope for this milestone:
- Multi-stance variants (Left/Right/High/Low) — added in follow-up once Top is proven.
- Real-time Kimodo inference inside the game loop — primitives are baked offline into `primitives.ron`.
- Weapon mesh/sword attachment — presentation-only, handled separately.

## Data Flow

### 1. Kimodo generation

Command shape:

```bash
kimodo_gen \
  "overhead longsword strike, two handed, planted feet" \
  --model Kimodo-G1-SEED-v1 \
  --duration 1.2 \
  --num_samples 4 \
  --seed 20260709 \
  --output data/kimodo/strike_longsword_top.npz
```

Each sample is saved as an NPZ with at least:
- `posed_joints` [T, J, 3]
- `local_rot_mats` [T, J, 3, 3]
- `root_positions` [T, 3]
- `global_root_heading` [T, 2]

### 2. Skeleton compatibility

Our MotionBricks feature pipeline expects the `G1Skeleton34` joint ordering (see `src/motion.rs`). Kimodo-G1 uses the Unitree G1 skeleton but may expose a different joint count/order. The converter must:

1. Map Kimodo joint indices to `G1Skeleton34` indices by name.
2. If Kimodo uses a superset (e.g. 51 joints), drop or ignore non-body joints and keep the 34 body joints.
3. If ordering differs, permute before feature computation.
4. Fail loudly if a required joint is missing.

### 3. Feature computation

Reuse the existing feature computation from `tools/extract_motion_features.py` (or its Rust equivalent) on the mapped G1 rotations/root. The output must be a `[T, 414]` float32 array matching the current `feature_dim` in `primitives.ron`.

Root-locking is applied in the converter so that the VQVAE does not drift, mirroring the `root_lock=True` fix applied to CMU clips.

### 4. Encoding

`tools/encode_primitives.py` is extended to accept Kimodo feature directories in addition to CMU `.npy` files. It writes entries to `assets/data/primitives.ron` with:

```ron
(
    action: Strike,
    weapon: Longsword,
    stance: Top,
    source_id: "kimodo_strike_001",
    feature_window: [...],
    root_target: (position: [...], heading: ...),
)
```

### 5. Action matrix update

`src/combat.rs` action profiles remain as constraint abstractions, but the runtime now requests MotionBricks clips by `(action, weapon, stance)` and the library supplies the Kimodo-backed primitive. `src/action_matrix.rs` timing data and matchup rules are updated to match the generated motion lengths.

## Prompt Design

Each action has a small prompt ladder (3-4 variants) so we can generate multiple candidates and pick the best by QA score. Examples:

| Action | Primary prompt |
|--------|----------------|
| Strike | "two handed overhead longsword strike downward, planted feet" |
| Block  | "high sword block, two handed longsword, brace and deflect" |
| Thrust | "forward longsword thrust, two handed, lunge step" |
| Grab   | "close quarters grapple, step in and grab torso, unarmed" |
| Dodge  | "side step evasion, sword guard, lean away from attack" |
| Idle   | "standing guard with longsword, relaxed ready pose, breathing" |

Prompts are stored in `tools/data/kimodo_prompts.json` under version control.

## Verification Plan

Every generated primitive must pass:

1. **Skeleton validation** — 34 G1 joints present and correctly mapped.
2. **Feature shape** — `[T, 414]` with T divisible by 4 (MotionBricks token boundary).
3. **Root lock** — `root_path == 0.0 m` after conversion.
4. **Velocity sanity** — max joint velocity within the same band as CMU Strike/Idle (~0.9–1.1 m/frame).
5. **Visual QA** — `tools/qa/visual_verify_primitives.py` produces readable stick-figure keyframes.
6. **Integration test** — `cargo test --test motion_service_integration` passes.

## Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| Kimodo-G1 joint ordering differs from G1Skeleton34 | Build an explicit name-based mapping and fail on mismatch. |
| Generated motion drifts or has foot skate | Root-lock and use Kimodo's built-in post-processing; reject bad samples. |
| Prompts produce inconsistent semantics | Generate 4 samples per prompt and select by QA; keep prompt ladder. |
| VQVAE decode fails on new feature distribution | Validate a decoded clip through `mb_probe` before encoding the full library. |
| License uncertainty for generated output | Use Kimodo-G1-SEED-v1 (NVIDIA Open Model) and CC-BY BONES-SEED annotations; document in `tools/data/mocap_manifest.json`. |
