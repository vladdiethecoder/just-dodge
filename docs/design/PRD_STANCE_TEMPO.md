# PRD: Stance and Tempo

## 1. Purpose

Provide readable, martial-arts-authentic pre-contact information through stance while gabling aggression and recovery through a tempo resource.

## 2. Invariants

- Stance is visible to the opponent as a MotionBricks-driven martial-arts pose cue but does not reveal exact hidden action.
- Tempo gates the next action selection; it never cancels a committed active action.
- Stance poses and transitions are generated by MotionBricks with authentic martial-arts weight and hip drive.
- Heavy armor and injury modify tempo and stance availability deterministically.
- Stance/tempo state is part of the truth hash.

## 3. Interface Contract

### Inputs
| Name | Type | Source | Description |
|---|---|---|---|
| desired_action | ActionId | PRD_INPUT.md | Action the player wants to select |
| desired_stance | Stance | PRD_INPUT.md | Stance the player wants to adopt |
| current_tempo | f32 | PRD_COMBAT_TRUTH.md | Current tempo resource |
| armor_state | ArmorState | PRD_ARMOR.md | Loadout mass and ROM restrictions |
| injury_state | InjuryState | PRD_INJURY.md | Capability modifiers |

### Outputs
| Name | Type | Consumer | Description |
|---|---|---|---|
| valid_actions | ActionId[] | PRD_COMBAT_TRUTH.md | Actions affordable in current tempo/stance |
| action_cost | f32 | PRD_COMBAT_TRUTH.md | Tempo cost of selected action |
| tempo_regen | f32 | PRD_COMBAT_TRUTH.md | Tempo recovered after exchange |

### Events / Signals
| Event | Payload | When Fired |
|---|---|---|
| stance_changed | { player_id, new_stance, frame_index } | Stance transition |
| tempo_exhausted | { player_id, frame_index } | Tempo dropped below action threshold |

## 4. Data Flow

1. At Plan phase, combat truth queries stance/tempo with desired action.
2. If tempo is insufficient, the action is rejected and the player must choose another or Disengage.
3. On resolve, tempo is consumed by the selected action and regenerated by a base amount plus stance/injury modifiers.
4. Disengage grants bonus tempo recovery.
5. Stance transitions are visible in the truth snapshot and drive presentation pose.

## 5. Control Flow

- **Who calls it:** PRD_COMBAT_TRUTH.md during Plan and Resolve phases.
- **Tick rate:** Per exchange; stance may transition during Observe.
- **Threading model:** Main thread.

## 6. Error Handling

- **Fail-open:** if an invalid stance/action pair is requested, combat truth defaults to Neutral/Disengage and logs the issue.
- **Degradation:** tempo UI shows a simplified indicator when exact numbers are disabled in Player mode.

## 7. Performance Budget

| Metric | Target | Worst Acceptable |
|---|---|---|
| Validation lookup | <1 µs | 10 µs |
| Tempo update | <1 µs | 10 µs |

## 8. Dependencies

- PRD_COMBAT_TRUTH.md — owns state machine and calls validation.
- PRD_INJURY.md — injury modifies tempo caps and regen.
- PRD_ARMOR.md — armor mass modifies costs and regen.

## 9. Open Questions

- Whether stance is selected per exchange or persists across exchanges.
- Whether tempo is shared or per-fighter.
- Visual/audio tempo affordance in Player mode.
